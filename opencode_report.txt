Executive summary
-----------------
Implemented the first clean v0 dataset foundation for the repository. The code establishes a small, standard-library-only base for raw episode metadata, processed dataset constants, measured TCP delta action computation, dummy raw data generation, raw episode validation, and unit tests.

The most important design decision is that the primary learning action label is measured TCP delta computed from robot state:

action[t] = TCP pose delta from t to t + 1

This makes the v0 dataset independent of the demonstration collection method. Hand-guided demonstrations can still produce a valid action label because measured robot state exists even when no commanded action exists. Teleoperation and future policy/shadow-mode logs can still preserve optional action streams, but those streams are not required by the initial learning pipeline.

Design rationale
----------------
The repository needs to support multiple possible demonstration sources: hand-guiding, keyboard or joystick teleoperation, SpaceMouse, VR controllers, and future shadow-mode policy logging. A dataset design that treats commanded actions as mandatory would exclude hand-guided demonstrations and would tie the dataset to one collection mechanism.

The v0 design therefore separates action streams into two categories:

1. Primary action label
   measured_tcp_delta, computed from measured robot TCP pose at consecutive frames.

2. Optional action-like streams
   commanded_action or commanded_twist from teleoperation, predicted_action from a shadow policy, and selected_action from a controller that chooses among streams.

The optional streams may be valuable later for debugging, analysis, behavior cloning variants, policy comparison, or studying operator intent. They are intentionally not required for the first processed dataset contract.

Dataset layer model
-------------------
Layer A is the raw dataset. It stores what happened during collection with minimal assumptions: metadata, robot TCP pose, joint states, wrench readings, camera frames, events, and any optional action streams that were available. The raw layer should preserve evidence and collection context.

Layer B is the processed dataset. It converts raw episodes into fixed per-frame arrays and labels for imitation learning. The v0 processed concept includes camera streams, robot state, wrench, joint state, a 25-dimensional model_state vector, and a 7-dimensional measured_action vector.

This separation is important because raw collection formats often change as hardware and teleoperation methods evolve. The processed format should remain stable enough for training experiments.

Measured action implementation details
--------------------------------------
The measured action code is in:

src/doosan_forcevla_data/convert/compute_actions.py

Public functions:
1. normalize_quat_xyzw(q)
2. quat_conjugate_xyzw(q)
3. quat_multiply_xyzw(q1, q2)
4. quat_to_rotvec_xyzw(q)
5. relative_rotvec_xyzw(q_t, q_t1)
6. compute_measured_tcp_delta_action(pos_t, quat_t, pos_t1, quat_t1, gripper_t=None, gripper_t1=None)

Input rules:
1. Positions must be list or tuple of length 3.
2. Quaternions must be list or tuple of length 4.
3. Quaternion convention is xyzw.
4. Numeric inputs must be finite real numbers.
5. NaN and infinity are rejected with ValueError.
6. Bool values are rejected even though bool is technically a Python numeric subtype.
7. Gripper values may be scalar floats or one-element list/tuple values.

Output rules:
1. Output is a plain Python list.
2. Output length is always 7 on success.
3. action[0:3] is pos_t1 - pos_t.
4. action[3:6] is relative rotation vector from quat_t to quat_t1.
5. action[6] is gripper_t1 - gripper_t if both gripper values are provided.
6. action[6] is 0.0 if either gripper value is missing.

Rotation convention:
1. Normalize q_t and q_t1 internally.
2. Compute q_rel = conjugate(q_t) * q_t1.
3. Convert q_rel to a rotation vector.
4. Use the shortest equivalent rotation by flipping quaternion sign if w < 0.
5. Use a small-angle branch for near-zero vector norm.

This convention matches the requested relative quaternion direction and avoids depending on scipy, numpy, ROS, transforms3d, or any other package.

Dependency and environment notes

The implementation intentionally uses only the Python standard library. Important standard-library modules used include argparse, csv, dataclasses, json, math, pathlib, tempfile, unittest, and typing.

No dependency fallback was needed beyond using PYTHONPATH=src for the uninstalled src-layout package. No pip, apt, conda, or external package installation commands were run.

Important behavior guarantees in v0
-----------------------------------
1. measured_tcp_delta can be computed without commanded actions.
2. commanded_twist.csv can exist for dummy or teleop-style episodes but is not the primary learning label.
3. quaternion input does not need to already be normalized.
4. invalid dimensions fail early with ValueError.
5. NaN and infinity inputs fail early with ValueError.
6. output action is a plain list of 7 finite floats.
7. raw validation returns clear error messages suitable for CLI display or tests.
8. dummy data is deterministic and small.

Known limitations and risks
---------------------------
1. No ROS recorder is implemented.
2. No dependency on the lab MyROS2 workspace is introduced.
3. No real Doosan controller interface is implemented.
4. No robot safety checks are implemented.
5. No camera calibration model is represented.
6. No camera timestamp synchronization validation is implemented.
7. No image decoding or image-size validation is implemented.
8. No raw-to-processed converter is implemented yet.
9. No LeRobot parquet export is implemented.
10. No Hugging Face integration is implemented.
11. No ForceVLA training code is implemented.
12. No schema version migration system is implemented.
13. The dummy commanded_twist stream is synthetic and should not be treated as a real robot command log.
14. The validator checks structural validity but not semantic correctness of a manipulation episode.
15. The wrench frame is recorded as sensor_or_tool_frame_as_recorded in the contract, so a future real dataset should explicitly document wrench frame and sign convention.
16. model_state includes ee_axis_angle, but the raw-to-processed converter that will compute it from ee_quat has not yet been written.
17. The current dummy image files are PPM placeholders, not actual RGB camera captures.
18. The package is not installed in the active environment, so commands currently need PYTHONPATH=src unless the project is installed later by the user.
